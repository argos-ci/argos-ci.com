## Context

Argos is a visual testing tool. The main output of a test is a change represented by a mask overlay, or more concretely a set of changed pixel coordinates. Most visual testing tools just output that change and stop there. Argos is a full visual testing platform that goes further.

We compute stats on change occurrences on auto approved builds, usually builds running on the main branch. That lets us detect flaky tests and allow users to ignore specific recurring changes to remove noise.

We also auto approve changes that were already approved on the same branch to reduce review time and keep the workflow fast.

All these features require one core capability: being able to tell that two changes are the same, or close enough to be treated as the same, across time.

## The problem

Our first implementation used the sha 256 of the mask image produced by our diffing library. If two masks had the same sha, then they were considered identical.

This is correct but too strict.

If a change contains 5000 pixels and only one pixel differs, the sha changes completely. Same if the change is slightly shifted. For a human reviewer it is the same change. For a hash, it is unrelated.

We needed something more tolerant. A kind of lazy hash for a change. A short stable string that stays identical when the change is globally the same, even if a few pixels differ.

## The solution direction

Input is a set of changed pixels. Output must be a short deterministic fingerprint string. Two visually equivalent masks should produce the same value so we can rely on a simple equality check and database indices.

This felt like a good candidate for an AI assisted implementation. I described the problem and asked for a performant approach to fingerprint diff masks in a tolerant way.

> I have some diffs in Argos that are represented by PNG images with red pixels, some diffs are very close, only a few pixels are different. I would like to create a fingerprint that is the same for two diffs that are very close to each other. What can I do in a performant way to do that?

The first answer proposed a pipeline that made sense:

1. Extract the red mask
2. Apply light morphology to absorb tiny noise
3. Normalize
4. Hash
5. Compare at a coarse scale

I asked for a TypeScript implementation. The code was clean and readable. The usage example focused on distance comparison between fingerprints, which was not exactly what I needed.

```ts
// Suppose you already decoded the diff PNG into RGBA.
const fpA = fingerprintDiffMaskFromRgba(rgbaA, widthA, heightA, {
  gridSize: 32,
  dilateRadius: 1,
});
const fpB = fingerprintDiffMaskFromRgba(rgbaB, widthB, heightB, {
  gridSize: 32,
  dilateRadius: 1,
});

if (isRoughlySimilarByOnes(fpA, fpB)) {
  const d = hammingDistanceFingerprint(fpA, fpB);
  const isNearDuplicate = d <= 40; // tune on your dataset
}
```

I clarified that I wanted a directly indexable string with equality semantics.

> Thing is, I want to be able to do fingerprintA === fingerprintB because I will put this in a database and I can't do custom comparaison, I need indices.

The refined approach was:

1. Build a binary mask of changed pixels
2. Optionally dilate by radius 1
3. Crop to bounding box
4. Split into a small grid, for example 16 by 16
5. Compute density per cell
6. Quantize densities into a few buckets
7. Hash the quantized grid into a fixed string

Small pixel level variations no longer change the fingerprint. Structural changes still do. That matched our needs.

## Fast validation with real diffs

I asked the AI to generate a test using real diff PNG fixtures from our codebase. I plugged it into the project and ran it on several masks we already knew were visually equivalent but not byte identical.

```ts
// The test generated by the AI compared two diff PNGs with the same fingerprint, even if they had some pixel differences.
import fs from "node:fs";
import path from "node:path";
import { PNG } from "pngjs";
import { describe, expect, it } from "vitest";

import { fingerprintDiffForEquality } from "../src/diff-fingerprint";

function readPngRgba(filePath: string): {
  rgba: Uint8Array;
  width: number;
  height: number;
} {
  const buffer = fs.readFileSync(filePath);
  const png = PNG.sync.read(buffer);
  return {
    rgba: png.data,
    width: png.width,
    height: png.height,
  };
}

describe("diff fingerprint", () => {
  it("compares diff-1.png and diff-2.png using an equality fingerprint", () => {
    const fixturesDir = path.join(__dirname, "fixtures");
    const diff1Path = path.join(fixturesDir, "diff-1.png");
    const diff2Path = path.join(fixturesDir, "diff-2.png");

    const diff1 = readPngRgba(diff1Path);
    const diff2 = readPngRgba(diff2Path);

    const fingerprint1 = fingerprintDiffForEquality(
      diff1.rgba,
      diff1.width,
      diff1.height,
      {
        gridSize: 16,
        dilateRadius: 1,
        // tweak these if needed
        densityThresholds: [0.002, 0.02, 0.08],
        padToSquare: true,
      },
    );

    const fingerprint2 = fingerprintDiffForEquality(
      diff2.rgba,
      diff2.width,
      diff2.height,
      {
        gridSize: 16,
        dilateRadius: 1,
        densityThresholds: [0.002, 0.02, 0.08],
        padToSquare: true,
      },
    );

    // This is what you want for DB indexing
    expect(fingerprint1).toBe(fingerprint2);
  });
});
```

After a bit of parameter tuning on grid size and density thresholds, the fingerprints matched exactly where we expected them to. That gave us a working baseline very quickly.

At this point we had a correct algorithm, but performance was the next constraint.

## Performance limits in TypeScript

This algorithm runs at the core of Argos, on every processed diff. Millions of times per month. Iterating over pixels and decoding PNGs in JavaScript is not ideal.

I ran a quick benchmark locally:

```
diff-A1.png: 35ms
diff-A2.png: 25ms
diff-A3.png: 12ms
diff-B1.png: 11ms
big-change.png: 981ms
```

Nearly one second on a large diff on a laptop CPU. Too slow for production workers.

## Porting the algorithm to Rust

Next step was obvious: same algorithm, Rust implementation.

I asked the AI to rewrite the algorithm in Rust.

> Can you rewrite this algorithm in Rust?

It generated a Rust implementation that I wasn't sure I could have written myself, especially the image processing part. I bootstrapped a project with napi and switched to Codex to integrate it into a Node binding. I do not write Rust daily, but the generated code was readable enough to review and adjust.

We kept the same fixtures and compared outputs between the TypeScript and Rust versions to ensure the fingerprint stayed stable.

Benchmark with the Rust version:

```
diff-A1.png: 4.5ms
diff-A2.png: 4.4ms
diff-A3.png: 4.4ms
diff-B1.png: 4.1ms
big-change.png: 393ms
```

Roughly three times faster on large masks, and consistently faster overall. Good enough for our workload.

I pushed the code to a dedicated repository, set up CI and publishing with AI help, and quickly had a ready to use npm package with Rust bindings.

## Running in production

I integrated the fingerprint into Argos and started computing it for every new diff to populate the database progressively.

After a few hours we hit crashes flagged by Sentry due to excessive memory usage. I asked the AI to optimize allocations and buffer handling. We iterated a few times on the Rust side to reduce memory pressure and avoid unnecessary copies.

We also switched to an async API to better fit our worker model and fixed edge cases like truncated PNG inputs.

Each iteration was reviewed and benchmarked again before rollout. We deployed progressively and monitored behavior before switching features to rely fully on the new fingerprint.

Today, our flaky detection and auto approval features rely on this fingerprinting algorithm implemented in Rust with AI assistance.

The project is open source by very tight to Argos, so if you are curious about the implementation details or want to use it in your own project, check it out: https://github.com/argos-ci/mask-fingerprint/

## Takeaway

AI did not replace the engineering work. It accelerated the algorithm design and the first implementations, especially in a language I do not use every day.

We still defined the constraints, reviewed the code, tuned parameters, measured performance, and hardened it in production.

Net result: a non trivial core algorithm designed and shipped much faster than usual, with full control kept on our side.
